0.8.12g
Release Notes

E 0.8.12g is an interim non-distributed release, but supporting a more modern version of the E language spec. It's primary motivation is to support work on compiling E, but it may also be preferred by those using E for non-distributed programming. The E installer is much improved on MSWindows, but these improvements have not yet been ported to other platforms, so this release is an MSWindows-only release. This is a very temporary situation -- the vast majority of E is fully platform independent.

As of this version E only support Java >= 2 (ie, a JDK/JRE >= 1.2).

All non-upwards compatibilities are highlighted in red.

Support for Locally Untrusted Installed Code

Originally we expected this to be one of that features we'd implement on the way to 1.0, as it required auditors, which are not currently supported, and it required a careful analysis of the Java API to determine what subset follows capability discipline. However, by taking some temporary shortcuts, this was a surprisingly painless change.

The plan

The plan for locally untrusted installed code remains as follows:

Code in "*.e" files evaluate in a scope which includes the magic powers that represent the authority given to the process as a whole by the host OS. (For conventional ACL-based OSes, this is normally all the authority associated with the user-account running the program.) This is called the privileged scope, and is analogous to the set of capabilities provided to the start program of the KeyKOS Big Bang. An example is "file__uriGetter", used in the expansion of the "<file:...>" syntax, which provides all the process's access to the host file system.

Code in "*.emaker" files evaluate in a scope which provides no authority, the universal scope. The only objects in the universal scope are clearly harmless ones such as "true", but not "file__uriGetter". The code in a "*.emaker" file evaluates to an emaker. This must happen under constraints which prevent the emaker from having and mutable state -- an emaker should be transitively immutable. The reason these are emakers is that these objects, having no authority or mutable state, are not generally very useful except for making other objects -- their yield. The messages asking an emakers to make its yield carries, as arguments, the authorities the yield should use. The emaker itself, lacking mutable state, cannot retain the provided authorities. The emaker, lacking mutable state, also cannot enable its separately-made yield to communicate. The emaker functions much like a KeyKOS Factory, and its yield are born confined.

  • Support for locally untrusted code (in *.emaker files)
  • New naming conventions
    • Observer pattern
      • Observer/Listener -> Reactor
      • whenFooDone(reactor), when reactor is only told once
      • addFooDoneReactor(reactor) / removeFooDoneReactor(reactor) when it's told repeatedly
      • reactor <- reactToFooDone(...) to let it know
    • Reference resolution states
      • fooPromise
      • farFoo
      • promiseFoo()
    • New names in Universal Scope (for *.emaker files)
      • pbc
      • sml`..`
    • New names in Privileged Scope (for *.e files)
      • <unsafe:...>
      • <swing:...>
      • <awt:...>
      • stderr (it already had stdout)
  • Improved concurrency abstractions in new "ref" package
    • RefHandler supports user-level writing of CapTP
    • New whenResolved and whenBroken suppresses multi-returns
    • New reactToLostClient for DeadManSwitch functionality
  • Experimental bundling of Waterken's Hydro library of capability-oriented collection classes.
  • Minimal-XML quasi-parser and DOM trees.
  • Improved system of guards and types
    • guard rejection now causes match failure rather than throwing
    • TypeDescs describe protocols
  • Syntax changes:
    • Extended multi-when-catch construct for gathering answers
    • "typedef" for making TypeDescs
    • Tyler's plurals on right side of operators
    • More liberal declaration syntax
      • "any" instead of "_" for region formation
      • :prim vs :order
      • function definitions now restricted to oNames
    • Lighter weight closure syntax using "_"
  • "e" driver improvements
    • "--pre" vs "-classpath"
    • Parameterized by eprops.txt
