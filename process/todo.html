Contributing to E
Requests for Hacking

I've been asked a few times how a technologist (hacker, theoretician, etc) might contribute to E. Here are some initial ideas.

Language Technology:

A precise statement of the semantics of Kernel-E, distributed Kernel-E, the core datatypes, or the network protocols. A proof that various security properties do in fact follow from these semantics.

A Kernel-E to J compiler.  The J in question should eventually be JVM bytecodes, but a plausible intermediate development step may be to translate E to Java, let the Java compiler take it the rest of the way.  The main constraining goal of such a compiler is not the traditional one, performance, but rather the passing through of source debugging info through to the *.class file, and hence the ability to reuse stock Java debuggers on mixed E and Java code.  (And to have Exception backtrace info give E source positions.)  When we want to pass E source line number info through to the *.class file, we'll probably need to give up on using Java source as an intermediate step in compilation.

Working out the precise interface between E's proposed auditors (a decentralized extensible notion of code verification) and the E verification framework, together with writing the first few critical auditors, such as for PassByCopy (audited for transparency and one-level immutability), confinement (audited for transitive immutability), and determinism.  As an example of what this will make possible: A system-provided memoizer that only accepts deterministic confined functions can itself be deemed confined, and therefore safely sharable, even though it uses shared mutable state (the memoization cache) to coordinate among its clients.

A standalone ENative -- to be the shell language for EROS.  As a result, the ENative plans have advanced since the web pages were put up, but no further progress has been made on implementation.  The current plans are to rewrite in C (rather than C++) in a way that works with the runtime conventions of JNI, such that it could run standalone or call and be called through JNI. To make the latency of accepting new E code low enough to be acceptable in a command line shell, we'd compile E to bytecodes and interpret the bytecodes, rather than, as explained on those pages, compile to C++.

Programming environment & tools written in E for supporting the E programmer, possibly starting with eBrowser and Elmer.  Most interesting would be a distributed debugger whose view of distributed computation was based on E's notion of partially ordered causality, rather than just treating each vat as an island.  I say "partially ordered" rather than "tree ordered" because the natural notion of a join is between a) the resolving of a promise and b) the messages sent to the promise, whose conjunction results in c) these messages being sent to the resolution of the promise.

We desperately need a good module system for breaking up E into separate files.  We've got one that works, but it doesn't seem very pleasant for large scale projects.  It also doesn't yet support a crucial feature supported by Java -- the creation of separate namespaces for resolving "import"s, so separately imported packages could stay out of each other's way.  This is necessary for multi-file mobile code. (Java's support of this through ClassLoaders is flawed in many ways, but it is a brave first step we should learn from.)

Static security annotations and checking, in the spirit of static type checking.  A great place to start is to wonder what static checks might have caught the recent MintMaker bug.  

Applications:

A capability-oriented desktop and secure user interface, in the spirit of, and perhaps an extension of EDesk and capDesk.

A multiway chat system (see EChat), but using the PetName Markup Language for secure human-designation of both people and transferred rights.

Some example smart contracts for, for example, more flexible trading of securities, or the Street Performer Protocol.
