Proxy Management in Neocosm

Last updated: [98/07/19 Bob] Author: Bob Schumaker.

Introduction

The Proxy Manager is the layer between raw communications and remote object references. High-level distributed object will never access data communication primitives directly. Instead, the higher layers will access remote objects through a proxy mechanism.

Requirements

  1. Encapsulate secure and authenticated remote object references in a 'simple' interface (currently Ref)
  2. Hide the details of the low-level communications mechanisms from the distributed object layer
  3. Provide consistent failure semantics for all proxies that travel over a single physical connection (i.e. if the underlying transport fails in a way that will affect all traffic over that connection, make sure that all clients of that connection also fail)

Architecture

Current Architecture Overview

In the current system, proxy objects are closely tied to the underlying connection in that the fragile references to objects go through a per-connection table (import/export).

Proposed Architecture Overview

How proxies get created, an overview (straw man until some of the actual details can be hammered out, success is assumed everywhere):
  1. FollowRef is invoked on a SturdyRef object (<registrar ID>, <object id>) [<object ID> is a swiss number]
  2. A proxy constructor is invoked (Proxy(<registrar ID>, <object ID>)
  4. The proxy code asks the low-level comm system for a connection object
  5. A connection is opened and returned (or a reference to an already open connection)
  6. The proxy code asks the connection for a reference to <object id>, specifying its local reference (this is optional really, since all messages could be addressed purely by swiss number -- but we're assuming a table structure along the lines of the current import/export table)
  7. The comm system makes the request of the remote machine
  8. The remote machine's comm system checks to see if <object id> is exported to the requesting machine
  9. The remote machine fills in its derived table and returns success to the requestor
  10. The local machine's comm system returns the appropriate data-comm level reference for the remote object (a table index, perhaps?)
  11. The proxy constructor finishes, and the proxy is now ready to accept 'E.Send' requests

The life cycle of a proxy, and how they are created and destroyed can be found in the Proxy Life Cycle document.
