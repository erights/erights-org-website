Introduction

This is a summary of guidelines for adding source code and packages into the source tree. The tree can be arranged in many ways. Experience has led to the understanding that a source tree should be neither too flat nor too nested. A tree that is too flat makes it difficult to locate an item. One that is too nested is difficult to navigate.

Requirements

  • Don't simply drop a package at the highest level of the tree. Take the time to figure out to what it is related. Use the subsystem diagram to help you figure out where you should be grouped.
  • The package structure should mirror the subsystem diagram as nearly as possible.
  • The directory structure should mirror the package name. This is crucial for compatibility with all standard IDEs. Remember that while ecomp allowed more than one public class per file, the standard java compiler does not.
  • If a package is from a third party it should be in the third party directory \tp
  • If a package is a library created by Communities.com but unrelated to other subsystem components, it should be placed in \lib
  • If a package is a utility that is generally useful, it should be placed in \util
  • If a package is useful only within a specific subsystem, it should be placed in \util below that subsystems directory
  • Similarly, test applications should be placed in \test if it is general test or in subsystem\test if it local to a subsystem.
  • If you have a large number of data classes that are currently stored in a single file consider moving them into a separate package. If this creates package access problems then make sure all of your data classes are easily distinguished as data classes.
  • If in doubt, send mail to ec_dev. Someone may have already given thought to a similar situation. Send email to me or amend this document when a convention has been reached so others will have knowledge of it.
